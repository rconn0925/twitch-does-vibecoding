<objective>
The vertical-slice core (BUILD-01): the build-session orchestrator drives a queued task through the full pipeline â€” enter BUILD_IN_PROGRESS, run the Sonnet research agent, run the Fable plan agent, re-screen the plan via COMP-02, then run the Fable build agent INSIDE the WSL2 sandbox (via `spawnClaudeCodeProcess` + the sandbox options), streaming stages through progress-events to the overlay build panel and chat narration, and return to IDLE on done. One build at a time (p-queue concurrency 1). The build query() is registered with an AbortController and the sandbox teardown so the Phase 1 veto can abort it. Crucially, COMP-02 runs BOTH before code is written (plan re-screen) AND during execution: each Write/Edit output batch is re-screened via `screenOutputBatch` (03-04), and a rejected batch aborts the build down the narrated compliance-failure path â€” the in-flight half of COMP-02, verified here, not a convention. Everything is injected (AgentRunner, SandboxAdapter, comp02, prompt-boundary) so vitest uses fakes â€” no real WSL2/query()/network â€” which is what makes the MVP happy-path e2e (03-02) go green here.
