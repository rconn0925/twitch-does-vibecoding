### Overlay build states (extends Phase 2's D2-18 state table)

The Phase 2 pill vocabulary is **locked** (D2-18: `STANDBY` / `VOTING OPEN` / `BUILDING` / `ON HOLD` are "the ONLY state words that ever reach the stream"). This phase adds **no new pill words** — the pill stays `BUILDING` for the entire research→plan→build pipeline; the new build-status panel carries the fine-grained stage narrative.

| Stage (from `progress-events.ts` vocabulary) | Pill | Build-status panel |
|----------------------------------------------|------|--------------------|
| `queued` | `BUILDING` (or prior mode) | Task still shown in the NEXT UP strip; build panel not yet mounted |
| `researching` | `BUILDING` | Panel mounts: task title + stepper, step 1 (Research) active |
| `planning` | `BUILDING` | Step 1 done, step 2 (Plan) active |
| `building` | `BUILDING` | Steps 1–2 done, step 3 (Build) active |
| `done` | `BUILDING` → flips to `STANDBY` after beat | 8-second celebratory "BUILT IT" beat (all steps green), then panel collapses |
| `failed` | `BUILDING` | Stepper freezes at current step; caption switches to amber non-alarming text. No error detail on stream. Retry/skip decision happens on the console |
| `refused` (D3-08, first-class) | `BUILDING` | Same non-alarming freeze; amber "moving on" caption. Narrated in chat, never shown as an error |

## Color

### Operator console additions — inherit Phase 1 palette UNCHANGED

`console.css` custom properties (`--bg #0F172A`, `--panel #1E293B`, `--accent #3B82F6`, `--destructive #DC2626`, `--status-approved #22C55E` green, `--status-held #F59E0B` amber, `--text #E2E8F0`, `--text-muted #94A3B8`) are reused as-is.

**Accent (`#3B82F6`) reserved-for list extended by exactly one element:** the **Retry Build** button (a forward/recovery action, same class as Phase 1's Resume / Discard-&-Resume). No other console accent change.

**Skip Task is NOT destructive-red:** skipping a failed/refused build is a routine live-pacing decision, not a ToS incident. Style it as a **neutral secondary button** (`--panel` background, `--text`, no accent, no red) so the scarce destructive red stays reserved for Halt / Veto / Reject (Phase 1). Veto Task itself is unchanged and remains destructive-red.

**Build-status pills on the console (informational, non-interactive — outside the 10% interactive-accent budget, per Phase 1's status-extension rule):**

| Build status | Value | Usage |
|--------------|-------|-------|
| Active stage (researching/planning/building) | `#3B82F6` (blue-500) dot | "in progress" stage dot — a status dot, not an interactive accent |
| Done | `#22C55E` (green-500) | Read-only "Build complete" pill |
| Failed / Refused (needs decision) | `#F59E0B` (amber-500) | Read-only "Needs a decision" pill — reuses Phase 1's held-for-review amber; distinct from the destructive-red Reject action |

Rationale: on the console the streamer needs the **honest** word ("failed", "refused") — but a failed *build* is not the same event class as a rejected *suggestion*, so it takes the amber "needs attention" treatment, keeping red exclusively for operator-initiated destructive actions and compliance rejections.

### Public overlay build panel — inherit Phase 2 overlay palette UNCHANGED

Page background stays fully transparent (OBS composites over video). The 60/30/10 split applies to rendered chrome:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `rgba(15, 23, 42, 0.88)` (slate-950 @ 88%) | Build-panel background |
| Secondary (30%) | `rgba(30, 41, 59, 0.92)` (slate-800 @ 92%) | Upcoming/inactive stepper step badges, stepper connector track |
| Accent (10%) | `#3B82F6` (blue-500) | **The current active pipeline-stage badge only** — the "where your eye goes" beat during a build |
| Text | `#F8FAFC` (slate-50) | All build-panel text |
| Success (semantic) | `#22C55E` (green-500) | Completed pipeline steps + the 8-second "BUILT IT" done beat (reuses Phase 2's winner green) |
| Non-alarming (semantic) | `#F59E0B` (amber-500) | The `failed`/`refused` frozen-stepper caption — honest but calm |

**Accent reserved-for (overlay build panel, explicit list):** the single currently-active pipeline-stage badge. Nothing else. (While `BUILD_IN_PROGRESS` the vote panel is hidden, so the accent's "one focal point" budget transfers cleanly from the leading candidate badge to the active build stage — no accent-budget conflict, since the two panels never render simultaneously.)

**Destructive `#DC2626` remains deliberately absent from the overlay** (carried forward from Phase 2). A build `failed`/`refused` shows amber + a calm caption, never red, never an error string, never the word "failed"/"error" on the public surface. Red on the broadcast surface reads as "the stream is broken."

**Legibility contract (hard rule, carried forward):** no overlay text renders directly over video — every text node sits on a Dominant-color panel. No text-shadow substitutes for a backing panel.

## Overlay Build-Status Panel — Layout Contract (PRES-02 / PRES-04, auto-selected)

Reuses the vote panel's slot and geometry (they never render at the same time — `VOTING_ROUND` vs `BUILD_IN_PROGRESS` are mutually exclusive modes).

| Property | Value |
|----------|-------|
| Position | Anchored lower-left, 48px from left and bottom edges (identical to the Phase 2 vote panel) |
| Size | 560px fixed width; height fits content |
| Background | Dominant `rgba(15,23,42,0.88)`, 12px radius, 24px padding |

**Panel anatomy (top to bottom):**
1. **Header** — `NOW BUILDING` (Heading 32/700). During the done beat it reads `BUILT IT`.
2. **Task title** — the chat-voted task text (Body 24/600), single line, **JS-truncated to 80 chars** + CSS `nowrap/ellipsis` backstop, rendered via **`textContent` only** (see safety rules). This is the one chat-derived string on this panel.
3. **Pipeline stepper** — three steps left→right: `Research → Plan → Build`, separated by connector tracks (Secondary). Each step is a badge + Label 20/600 caption:
   - **Completed** step: green (`#22C55E`) badge, slate-50 label.
   - **Active** step: accent (`#3B82F6`) badge, slate-50 label — the panel's only accent use.
   - **Upcoming** step: Secondary badge, muted (`#94A3B8`) label.
4. **Stage caption** — one line, Label 20/600 muted, drawn from a **fixed orchestrator-authored set** (never chat-derived, so always safe): `researching` → "Digging into the idea", `planning` → "Drafting the build plan", `building` → "Writing the code", `done` → "Live on screen now", `failed` → "Regrouping…" (amber), `refused` → "Skipping this one" (amber).

**Done beat (auto-selected, mirrors the Phase 2 winner beat):** on `done`, all three steps turn green, header → `BUILT IT`, caption → "Live on screen now"; hold **8 seconds**, then the panel collapses as the pill flips to `STANDBY`.

**Failed / refused:** the stepper freezes at the current step (no red, no new step), header unchanged, caption switches to the amber non-alarming text. The panel stays until the streamer's console decision resolves the build (retry re-animates the stepper; skip collapses the panel).

**Motion & update cadence (carried forward from Phase 2):**
- Stage transitions arrive as **state diffs** on the existing full-state-on-connect + diff channel (low frequency — a handful per build), never per-frame streaming.
- The active-badge fill animates with `transition: background-color 300ms ease-out` (same easing family as the Phase 2 tally-fill) so stage advances read as a deliberate beat, not a flicker.
- On ws disconnect the panel **freezes its last render and retries silently with backoff**. No error text ever renders on the broadcast surface. OBS scene-switch reloads take the full-state-on-connect path.

**Chat-derived text safety (hard rules, extend the Phase 2 invariant to the build panel):**
- The task title renders via **`textContent` only — never `innerHTML`**. The single-funnel zero-innerHTML scan must cover the build-panel code path in `overlay/public/`.
- Task title truncates to **80 characters** JS-side (append `…`) + CSS `white-space: nowrap; overflow: hidden; text-overflow: ellipsis` backstop.
- No user-controlled URLs, images, or markup are interpreted — the title is opaque plain text.

## App-Under-Construction Preview — Layout & Interaction Contract (PRES-03 / D3-12, auto-selected)

A **separate** page (`src/preview/public/`) served on its own route, pointed at by a dedicated OBS Browser Source. It is NOT the overlay and NOT the console.

**Isolation contract (hard rules — SAND-02 / D3-12):**
- The preview page holds **no WebSocket/HTTP connection to the orchestrator** and subscribes to no orchestrator state. Its only dynamic input is whether the sandboxed dev server (`http://localhost:<fixed-port>`, per 03-RESEARCH §d) is reachable.
- It renders **zero chat-derived text** and **zero orchestrator state** (no task title, no pipeline stage, no queue) — those live on the overlay. The preview shows only the live app + a static cosmetic label.
- The embedded frame's `src` is the dev-server URL and nothing else; the page exposes no host, orchestrator, or console surface.

**Layout:**

| Block | Position | Size |
|-------|----------|------|
| **Chrome status bar** | Fixed top edge, full width | 48px tall, Dominant `#0F172A` background |
| **App frame** | Fills the entire viewport beneath the status bar | `<iframe>` pointed at the dev server, background `#0F172A` (dark, so reload flashes never flash white) |
| **Waiting placeholder** | Centered over the app-frame region, shown only when the dev server is unreachable | Secondary `#1E293B` card |

**Status-bar anatomy:** left — title `APP UNDER CONSTRUCTION` (Body 24/600); right — a reachability indicator = a status dot + label (Label 20/600):
- Dev server reachable → green dot + `LIVE`.
- Dev server unreachable → amber dot + `STANDING BY`.

**Auto-refresh interaction (PRES-03):**
- While LIVE, the wrapper reloads the app frame every **5 seconds** (auto-selected; a cosmetic cadence knob, tune freely) so viewers see build progress without manual reloads. Recommended: double-buffer (load into a hidden frame, swap on `load`) to avoid a visible white flash; the dark frame background is the fallback if double-buffering is skipped.
- Reachability is polled on a short interval (auto-selected ~2s) via the frame's `load`/`error` events or a lightweight ping to the dev-server port. On an unreachable→reachable transition, the frame is (re)loaded and the placeholder hides.

**Waiting / empty / failure states (all calm, all non-alarming — never a raw browser error page on stream):**

| Condition | Placeholder headline (Heading 32/700) | Sub-line (Body 24/600 muted) |
|-----------|----------------------------------------|------------------------------|
| Dev server not up yet (first build spinning up) | **"Setting the stage…"** | "The build environment is starting up." |
| Between builds (previous done, next not started) | **"Between builds"** | "Next one's coming right up." |
| Build failed / dev server crashed | **"Reworking this one"** | "Back in a moment." |

The preview **never** shows a stack trace, an HTTP error code, a browser "can't reach this page" screen, or the color red. If the dev server is unreachable for any reason, the amber `STANDING BY` dot + the appropriate calm placeholder is the only thing that renders.

### Bot chat narration (D3-08 refusals + pipeline events — chat copy is UI copy)

Follows the Phase 2 rate-budget doctrine (**transitions → chat, high-frequency state → overlay**) and Phase 2's game-show-beat template style. All messages go through the single rate-limited sender queue. **Task titles inside chat messages truncate to 60 characters** (Twitch 500-char cap). Category labels come from the viewer-safe `CATEGORY_META` labels in `src/compliance/categories.ts` — chat never sees internal codes or classifier rationale.

Stage transitions are **low-frequency, one message per transition** (a handful per build) — well within the rate budget. Per-token/per-file build churn stays OFF chat (it belongs to the preview + overlay).

| Event | Template |
|-------|----------|
| Build picked up (task → researching) | `Building "{title}" now — researching how to pull it off.` |
| Stage → planning | `Plan's coming together for "{title}"…` |
| Stage → building | `Writing the code for "{title}" now — watch it come alive.` |
| Build done | `"{title}" is built — it's live on screen. GG.` |
| Model refusal mid-build (D3-08, first-class) | `Heads up — the build agent won't build "{title}". Moving on to the next one.` |
| Build failed, auto-retrying once (D3-09) | `"{title}" hit a snag — giving it one more shot.` |
| Build failed, streamer deciding (D3-09) | `"{title}" won't build cleanly — streamer's calling retry or skip.` |
| Streamer chose retry | `Another go at "{title}" — here we go.` |
| Streamer chose skip | `Skipping "{title}" — on to the next idea.` |
| COMP-02 plan rejected (D3-06) | `"{title}" didn't pass the second safety check — can't build that one. Next up.` |
| Streamer veto aborts in-flight build (D3-10) | `Build stopped — pulling the plug on "{title}". Standing by.` |
