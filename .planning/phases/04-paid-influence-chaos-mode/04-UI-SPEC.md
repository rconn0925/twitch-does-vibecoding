# Phase 4 — UI Design Contract

> Visual and interaction contract for **paid influence & chaos mode**: the public OBS overlay gains a free-reign-window panel and a chaos-mode indicator (PAID-01/02/04, CHAOS-01), and the operator console gains a chaos-mode toggle and a control-window panel with a Revoke Window control (PAID-04). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**This is an EXTENSION of `03-UI-SPEC.md`, not a new design system.** Every token, type scale, spacing value, and hard rule declared there (60/30/10 palette, locked pill vocabulary discipline, no-red-on-broadcast, amber-non-alarming, `textContent`-only + truncation for chat/donor-derived text, full-state-on-connect + diffs, 44px console click targets) carries forward unchanged. This document declares only the deltas: two new mutually-exclusive overlay panels, a controlled two-word extension to the locked pill vocabulary, and three new console controls.

### Mode-driven panel exclusivity (extends Phase 3's "panels share a slot" pattern)

The existing `StreamMode` enum already carries `FREE_REIGN_WINDOW` and `CHAOS_MODE` as `IDLE`-reachable states with a `BUILD_IN_PROGRESS` exit (Phase 1). The overlay's lower-left slot therefore now has **four** mutually-exclusive occupants, exactly one of which renders per the current `mode` — no client-side precedence logic needed; the state machine already guarantees only one mode is active at a time (D-05: paid window > chaos > vote):

| `mode` | Lower-left slot occupant | Pill word |
|---|---|---|
| `IDLE` | none (silent absence) | `STANDBY` |
| `VOTING_ROUND` | vote-panel (Phase 2, unchanged) | `VOTING OPEN` |
| `FREE_REIGN_WINDOW` | **free-reign-panel (NEW)** | **`FREE REIGN`** (NEW) |
| `CHAOS_MODE` | **chaos-panel (NEW)** | **`CHAOS MODE`** (NEW) |
| `BUILD_IN_PROGRESS` | build-panel (Phase 3, unchanged) | `BUILDING` |
| `HALTED` | none (silent) | `ON HOLD` |

**Pill vocabulary extension (controlled amendment to D2-18):** Phase 2 locked the pill vocabulary to `STANDBY`/`VOTING OPEN`/`BUILDING`/`ON HOLD` — one word per top-level `StreamMode`. Phase 4 adds two genuinely new top-level modes that are themselves stream-worthy spectacle moments (a donor bought guaranteed control; chaos is about to pick something at random) — following the SAME one-word-per-mode rule already established, not inventing a new pattern. The vocabulary is now six words, still a fixed, orchestrator-authored, never-chat-derived set. No further pill words are added by this phase.

## Overlay — Free-Reign-Window Panel (PAID-01/02/04, NEW)

Reuses the vote/build panel's exact slot and geometry — anchored lower-left, 48px from left/bottom, 560px fixed width, Dominant background, 12px radius, 24px padding. Mounted only while `mode === FREE_REIGN_WINDOW`; unmounts the instant `mode` changes (expiry or revoke — both look identical on the overlay, see Copywriting).

**Panel anatomy (top to bottom):**
1. **Header** — `FREE REIGN` (Heading 32/700, static, never chat-derived).
2. **Controller display name** — the donor's (or channel-points redeemer's) display name (Body 24/600), single line, **truncated to 32 characters** (shorter cap than the 80-char task title — this is a name field, not a sentence) + CSS `nowrap/ellipsis` backstop, rendered via **`textContent` only**. This is untrusted donor/chat-derived text — same XSS discipline as the build panel's task title (T-03-15 lineage).
3. **Countdown** — `m:ss`, Display 48/700, `font-variant-numeric: tabular-nums`, computed client-side from an absolute `endsAtMs` on a 1s tick (identical mechanism to the vote panel's countdown — never server-streamed per-frame). Flips to Accent blue normally; flips to **Urgency amber** in the final 10 seconds (reuses `.countdown-final` verbatim, same `FINAL_COUNTDOWN_MS = 10000` constant).

**No task title, no instruction text, no trigger type, no raw amount ever appears on this panel.** Once a donor's instruction clears the gate and is queued, `mode` moves to `BUILD_IN_PROGRESS` and the existing build panel takes over the slot exactly as it already does for a vote winner — no new build-panel behavior needed.

**States:**

| State | Panel behavior |
|---|---|
| No active window | Panel absent (silent — `mode !== FREE_REIGN_WINDOW`) |
| Active, > 10s remaining | Accent-blue countdown |
| Active, ≤ 10s remaining ("expiring") | Amber countdown (`.countdown-final`) |
| Expired naturally | Panel unmounts on the next state push — no special beat, no error text |
| Revoked by streamer | Panel unmounts on the next state push — visually identical to natural expiry; the overlay never distinguishes "why" a window ended (that detail lives on the console/audit only) |

## Overlay — Chaos-Mode Panel (CHAOS-01, NEW)

Same slot, geometry, and background as the free-reign panel. Mounted only while `mode === CHAOS_MODE`. **No countdown** — chaos mode is a persistent streamer toggle, not a timed window, so this panel has no expiry concept.

**Panel anatomy:**
1. **Header** — `CHAOS MODE` (Heading 32/700, Winner-green accent treatment on the header text or an adjacent status dot — the panel's only accent-equivalent use, deliberately green not blue).
2. **Caption** — one fixed line, Label 20/600, muted: **"Picking the next build at random"**. Static, orchestrator-authored, never chat-derived (same fixed-caption pattern as the build panel's stage captions).

**States:**

| State | Panel behavior |
|---|---|
| Chaos mode off | Panel absent (silent) |
| Chaos mode on, awaiting next pick | Panel shown, static (no countdown, no motion beyond the header's existing 300ms color-transition family if the badge state changes) |
| Chaos pick made → queued | `mode` moves to `BUILD_IN_PROGRESS`; panel unmounts, build panel takes over, exactly as with a vote winner or a free-reign instruction |

## Overlay — Implementation Note (non-binding guidance, matches existing precedent)

Following the exact technique Phase 3 used to add the build panel without touching `index.html` (see `overlay.js`'s `document.createElement("section")` pattern), the free-reign panel and chaos panel should be created the same way at runtime and appended once — no changes to the three static blocks (`state-pill`, `vote-panel`, `queue-strip`) already in `index.html` are required. `renderAll()` gains two more render functions (`renderFreeReignPanel()`, `renderChaosPanel()`) following the same `replaceChildren()` + `hidden` toggling shape as `renderBuildPanel()`.

## Console — Chaos Mode Panel (CHAOS-01, NEW)

New `.panel` section, visible whenever not `HALTED` (same visibility rule as the existing round-panel/build-panel — hidden during triage takeover).

**Anatomy:**
- **Title** — "Chaos Mode" (Heading 20/600).
- **Status pill** — "ON" (`.status-pill.status-approved`, green) or "OFF" (plain muted text, no pill — consistent with how the console only pills genuinely notable states).
- **Explanatory body** (Body 16/400, fixed copy): *"When on, the next task is picked at random from the approved pool instead of a vote. Every pick still passes the same compliance check — chance never touches payment."* (carries the CHAOS-02 compliance framing into console copy).
- **Toggle button** — single button, label and style flip with state (no modal, single click, matches the console's zero-confirmation-friction discipline, D-03):
  - OFF → **"Turn On Chaos Mode"**, `.button-accent`.
  - ON → **"Turn Off Chaos Mode"**, `.button-neutral`.

## Console — Control Window Panel (PAID-01/02/04, NEW)

New `.panel` section, positioned directly after the existing build-panel (before round-panel) — this panel governs "what determines the next build" alongside the round panel, so it sits in that same visual group. Visible whenever not `HALTED`.

**Empty state (no active window):**
- Heading: **"No active window"** (Heading 20/600).
- Body: **"Donations and channel-points redemptions open a free-reign window here."** (Body 16/400, matches the existing empty-state copy shape used elsewhere on this console).

**Active state — full detail (console MAY show what the overlay must not):**

| Field | Label (14/600 muted) | Value treatment |
|---|---|---|
| Donor / redeemer | "Donor" | Full display name, **no truncation** — `textContent` + `overflow-wrap: anywhere` (same treatment as `.card-text`/suggestion text elsewhere on this console), soft 120-char safety backstop against layout blowout |
| Trigger type | "Trigger" | Read-only `.status-pill` badge, neutral styling (`background: var(--panel); border: 1px solid var(--text-muted); color: var(--text);` — informational, not a decision status): **"Donation"** or **"Channel Points"** |
| Amount | "Amount" | Donation: `$X.XX` (formatted from stored cents). Channel points: `N pts`. **Console-only — never rendered on the overlay** (PAID-04's raw-amount-stays-off-broadcast rule) |
| Time remaining | "Time remaining" | `m:ss`, `tabular-nums`, 1s client tick from `endsAtMs` (same mechanism as the round panel's own countdown). Flips to `--status-held` amber in the final 10 seconds — mirrors the overlay's own urgency threshold |
| Compliance framing | — | Fixed body line (Body 16/400): **"{Donor} has guaranteed selection for their next instruction — it still has to clear the same compliance check as everything else."** Never implies payment bypasses the gate. |
| Action | — | **"Revoke Window"** button, `.button-caution` (amber-bordered, NOT red, NOT accent — see Color), 44px min height, single click, no modal |

**After Revoke Window is clicked:** single click, no confirmation modal (matches Veto/Skip's existing zero-modal-friction discipline). An optional one-tap reason-tag row appears immediately after, reusing the existing D-18 generic reason-tag component (`ToS risk` / `boring` / `too big` / `gut feeling` / `other` / skip) with a new `reasonContext.kind = "revoke-window"`. A transient confirmation line appears once (same pattern as the veto-abort confirmation line, `.build-abort-line`-style: `color: var(--status-held); font-weight: 600;`): **"Window revoked — {donor}'s free reign ended early. Back to normal."** — cleared on the next render, same lifecycle as the existing `roundStartError`/`dev-feedback` transient-message pattern.

**States:**

| State | Panel behavior |
|---|---|
| No active window | Empty state (above) |
| Active, > 10s remaining | Normal detail table, default text color for time |
| Active, ≤ 10s remaining ("expiring") | Time-remaining value turns `--status-held` amber |
| Window expired naturally | Reverts to empty state on the next ws push; no special transient line (expiry is routine, not an operator action — nothing to confirm) |
| Window revoked by streamer | Transient amber confirmation line (above), then reverts to empty state |

## Truncation Rules (extends Phase 2/3's chat-derived-text discipline)

Donor/redeemer display names are **untrusted, chat/donor-derived text** (StreamElements `displayName` is donor-set on an external platform, Twitch `user_name` is chat-derived) — the identical `textContent`-only + truncation discipline that already governs task titles and queue chips applies here.

| Surface | Field | Max length | Method |
|---|---|---|---|
| Overlay free-reign panel | Controller display name | **32 chars** | JS `truncate()` + `textContent` + CSS `nowrap/ellipsis` backstop |
| Console control-window panel | Donor / redeemer name | **no hard truncation** (full detail permitted); 120-char safety backstop via `overflow-wrap: anywhere` (matches `.card-text`) | `textContent` only |
| Chat narration | Donor / redeemer name | **25 chars** (a Twitch-handle-scale cap; StreamElements names can run longer, so this is a genuine truncation, not just a backstop) | Same `truncate()` helper used for the existing 60-char task-title chat cap |
| Chat narration | Task title (unchanged) | **60 chars** | Existing Phase 3 rule, unchanged |
| Overlay free-reign/chaos panels | Task title | **not shown** — these panels never render a task title; the build panel (unchanged, 80-char cap) takes over once an instruction is queued | — |

No raw donation amount, redemption cost, or currency figure is ever rendered on the overlay — this is a content rule, not a truncation rule (see Color / Public overlay above).

### Bot chat narration (new events, follows the Phase 2/3 never-silent + rate-budget doctrine)

Stage transitions here are low-frequency (a handful per window/toggle), well within the existing rate budget. Donor/redeemer names truncate to 25 chars; task titles (where present) truncate to 60 chars — both via the existing single rate-limited sender queue.

| Event | Template |
|-------|----------|
| Free-reign window opened (donation) | `{name}'s donation just bought free reign — what should we build?` |
| Free-reign window opened (channel points) | `{name} redeemed free reign — what should we build?` |
| Instruction queued during window (PAID-03) | `Building "{title}" next — {name}'s pick, guaranteed selection (still passed the safety check).` |
| Instruction rejected/held during window | `{name}'s free-reign pick didn't clear the safety check — window's still open, try another instruction.` |
| Window expired naturally | `{name}'s free reign window has ended. Back to the normal loop.` |
| Window revoked by streamer | `Streamer ended {name}'s free reign window early. Back to the normal loop.` |
| Donation/redemption arrives mid-window (queued behind cooldown, never silently dropped, D-05) | `Got it, {name} — another window's active right now. Yours is next in line.` |
| Donation/redemption arrives during cooldown | `Thanks, {name} — you're on cooldown a bit longer. No window this time, but keep watching.` |
| Chaos mode toggled on | `Chaos mode is ON — the next pick is random from the approved pool, no vote.` |
| Chaos mode toggled off | `Chaos mode is OFF — back to voting for the next build.` |
| Chaos pick selected | `Chaos picked "{title}" at random — building it now.` |
