## Scope & Surface

One surface, and it is genuinely **NEW** — but it is a document/list page read up close in a normal browser tab, not a broadcast overlay composited over video and not a private operator tool. It sits in a fourth posture alongside the three shipped ones.

| Surface | Posture | Governing rules |
|---------|---------|-----------------|
| **Build-history changelog** (`src/history/public/`, new) | Served, read-only, full-list-on-load. Audience-facing AND explicitly "safe to screen-share" (D-04) — read at normal reading distance in a browser tab, not transcoded video. Zero interactive elements except pagination ("Load older nights"). | This spec, below. Extends the shipped design system; invents zero new colors and zero new type scale. |

**Why this surface borrows the console's type scale, not the overlay's:** the overlay's 20/24/32/48px scale exists solely because nothing smaller survives Twitch's 1080p→720p transcode over moving video (Phase 2/3 rationale). The changelog is a normal static HTML document rendered at 100% in a browser window — it has the same reading conditions as the operator console, not the overlay. Reusing the console's 14/16/20/28 scale is the "invent nothing new" choice.

**Why this surface borrows the console/preview dark palette, not a light one:** every shipped surface in this product (overlay, preview, console) is dark-slate. A changelog in a different visual language would break the "this is one product" read the moment someone screen-shares it. Reuse `#0F172A` / `#1E293B` / `#F8FAFC` as-is.

**Untouched surfaces:** overlay, preview, and console are unchanged by this phase — no delta, no re-spec.

## Layout Contract

### Page structure (top to bottom)

| Block | Content |
|-------|---------|
| **Page header** | Title **"Build History"** (Display 28/600) + tagline **"The audience built this together."** (Body 16/400, muted) |
| **Night groups** | Reverse-chronological (newest stream night first). Each group = a heading + its entry rows. See below. |
| **Load-older control** | A single centered button, shown only when more history exists beyond the initially-loaded set (`hasOlder: true`). Absent entirely (not disabled) once history is exhausted — mirrors the shipped "hidden entirely, not empty" convention from the overlay/console panels. |
| **Empty state** | Replaces night groups + load-older entirely when there is no history at all. See Copywriting. |

### Night-group anatomy

```
┌─────────────────────────────────────────────┐
│ Tuesday, July 8, 2026              3 builds  │  ← Heading 20/600 + Label 14/600 muted count, baseline-aligned row
├─────────────────────────────────────────────┤
│ [chip] Suggestion title text here…   ● Built  9:42 PM │  ← entry row
│ [chip] Another suggestion title…   ● Refused  9:15 PM │
│ [chip] A third suggestion…          ● Failed  8:58 PM │
└─────────────────────────────────────────────┘
```

- Night groups stack with `lg` (24px) gap between groups.
- Night-group heading row: flex, `space-between` — night label left (Heading 20/600), entry count right (Label 14/600, muted, e.g. "3 builds").
- A 1px hairline divider (`Secondary` color, full width) sits between the heading row and the first entry row (auto-selected — a lightweight visual separator, not a new color).

### Entry-row anatomy (D-05: title + provenance chip + result state + timestamp)

Single-line flex row, `md` (16px) gap, `Secondary` background, 12px radius, `md` (16px) vertical / `lg` (24px) horizontal padding, `sm` (8px) vertical gap between rows within a group:

| Element | Position | Style |
|---------|----------|-------|
| Provenance chip | Leftmost, fixed width | See Provenance Chip Mapping below |
| Suggestion title | Flex-grow, min-width 0 | Body 16/400, `textContent` only (untrusted, chat-derived), single line |
| Result-state badge | Fixed, right of title | 8px dot + Label 14/600, per Result-State Treatment below |
| Timestamp | Fixed, rightmost | Label 14/600, muted, `font-variant-numeric: tabular-nums` for column alignment down the list |

**Truncation (hard rule, extends the Phase 2/3/4 dom-safety invariant):** the suggestion title is **JS-truncated to 100 characters** (append `…`) + CSS `white-space: nowrap; overflow: hidden; text-overflow: ellipsis` backstop. 100 chars (not the overlay/console's 60/80) because this page has more horizontal room and no video-encode legibility pressure — but a hard cap still exists so one adversarially-long approved suggestion title can never break the row layout. Rendered via **`textContent` only, never `innerHTML`** — the zero-innerHTML source scan must cover the new `history.js` code path.

## Provenance Chip Mapping (D-05, reuses the Phase-4 vocabulary)

Reuses the exact `.provenance-chip` visual pattern from `overlay.css` (Secondary background, 4px 12px padding, 8px radius) at the console-scale Label size (14/600 instead of the overlay's 20/600).

| Ledger `provenance` value | Chip text | Chip text color |
|---------------------------|-----------|------------------|
| `vote` | **VOTE** | `#F8FAFC` (default — neutral, reuses the overlay's "chaos gets no color" pattern for the default case) |
| `paid` | **FREE REIGN** | `#8B5CF6` violet — reused exactly, Phase 4's reserved-for list extended by exactly this one additional surface |
| `chaos` | **CHAOS PICK** | `#F8FAFC` (default — identical treatment to the overlay's chip, "chaos gets no color") |

**Deliberate difference from the overlay:** on the live overlay, a vote-winner build gets **no chip** (the default, implicit case, since the vote panel context already makes it obvious). On the changelog, every entry is decontextualized from the live show, so **`vote` gets an explicit chip too** — a viewer scrolling history has no other cue for how a given build got picked. This is the one small, deliberate extension beyond a literal copy-paste of the Phase 4 vocabulary; it adds a label, not a color (zero new color tokens).

**Copy-separation invariant (carried forward, hard rule):** the `FREE REIGN` chip never mentions money/amount/donor identity; the changelog entry carries no donor name, no amount, and no trigger-type distinction between donation vs. channel-points (both collapse into `paid` — D-01's `provenance` vocabulary is already the coarse three-way split, and this page goes no finer than that). This is a deliberate coarsening beyond even the overlay banner (which does show a truncated donor name for the ~90 seconds a window is live) — a changelog is a long-lived public record, so donor identity is dropped entirely rather than persisted indefinitely on a screen-shareable page. (Claude's-discretion call, consistent with CLAUDE.md's "no donor financial detail" / coarse-projection doctrine; flag for streamer confirmation if a per-donor "hall of fame" is wanted later — that would be a new, explicit feature, not a default.)

## Result-State Treatment (D-03, honest outcome — never alarming)

| Ledger `result` value | Dot + label | Color | Copy |
|------------------------|--------------|-------|------|
| `built` | ● Built | `#22C55E` green | **"Built"** |
| `refused` | ● Refused | `#F59E0B` amber | **"Refused"** |
| `failed` | ● Failed | `#F59E0B` amber | **"Failed"** |

**Hard rules (D-03, load-bearing, carried forward from the overlay's "no red" doctrine):**
- Refused and Failed are visually identical in color (both amber) — the distinction is in the WORD only, never in alarm-level color. Neither is ever red.
- These are the ONLY three result words that ever render on this page. No other outcome vocabulary (e.g. "error", "crashed", "rejected") may appear.
- A `refused`/`failed` entry's suggestion **title still renders** (it already passed the intake gate — D-03) — what never renders anywhere on this page is a pre-gate or banned/rejected-at-intake suggestion. That filtering happens at the query layer (the changelog reads only gate-APPROVED entries by construction); this UI-SPEC's contribution is: the page has no code path that could ever receive or render disallowed text, because the wire only ever carries the three fields in the State Shape below.

## Pagination / Scroll-Cap (D-04)

**Chosen pattern: paginated by stream night, "Load older nights" button (auto-selected).** Not infinite scroll (no scroll-position bookkeeping needed for a low-frequency, single-streamer v1 surface) and not a full scroll-cap-with-clip (would hide real history behind a fixed-height box, which reads as broken on a "shareable" page).

- **Initial load** serves the **10 most recent stream nights** (auto-selected page size) as the full-state-on-load payload (D-04's "full-state-on-load" pattern, adapted from the WS surfaces to a plain HTTP GET since this page has no live-update requirement — see State Shape below).
- **"Load older nights"** button (accent, 44px min height, single click, no modal) fetches the next 10 nights and appends them to the bottom of the list. Disabled state is never shown — the button is simply absent (`hidden`) once `hasOlder: false`.
- **Per-night entry cap (defensive, auto-selected):** if a single stream night somehow has more than 50 entries, the group renders the first 50 (newest-first within the night) plus a static muted line **"+{N} more that night"** (no further pagination within a night — a night with >50 builds is far outside any realistic v1 stream, this is a defensive ceiling only, not a designed-for interaction).

## Served-Page State Shape (for the planner — coarse, no host internals, no donor financial detail)

Reuses the "full-state-on-load" doctrine from the WS surfaces, adapted to a plain paginated HTTP GET (no WebSocket needed — this page has no live-tick/live-update requirement; a viewer who wants the newest entry reloads the page, same as any changelog).

```
GET /history                     → serves the HTML shell (history.html) + history.css + history.js
GET /api/history?before={nightKey}&limit=10   → HistoryPage JSON (first call omits `before`)

HistoryPage {
  nights: HistoryNight[];
  hasOlder: boolean;              // true if a further ?before= call would return more nights
}

HistoryNight {
  nightKey: string;               // session/day grouping key (D-02), e.g. "2026-07-08" — server-derived, opaque to the client
  nightLabel: string;             // human display, e.g. "Tuesday, July 8, 2026" — SERVER-FORMATTED (no client-side timezone/date math)
  entries: HistoryEntry[];        // newest-first within the night
  entryCountLabel: string;        // e.g. "3 builds" — server-formatted plural, avoids client pluralization logic
}

HistoryEntry {
  buildId: string;
  title: string;                  // gate-APPROVED suggestion text ONLY (D-03) — untrusted, textContent-only, JS-truncated to 100 chars client-side
  provenance: "vote" | "paid" | "chaos";     // D-01 vocabulary — coarse three-way split, no trigger-type or donor detail
  result: "built" | "refused" | "failed";    // D-03 honest outcome, fixed three-word vocabulary
  timeLabel: string;               // server-formatted "9:42 PM" — no client timezone math
}
```

**What this wire explicitly excludes (coarse-projection discipline, carried forward from Phases 3/4):** donor identifier, donation/redemption amount or cost, trigger-type detail (donation vs. channel-points), any classifier rationale or category label, any pre-gate/rejected/banned suggestion text, any host/orchestrator/sandbox internals, build duration, and any per-file/per-token build detail. The changelog is a public record of outcomes, not an operational log — the audit ledger (console-only, streamer-facing) remains the place for full detail.

**Chat-derived text safety (hard rule, extends the existing invariant to `history.js`):** `title` renders via `textContent` only — never `innerHTML`. The zero-innerHTML source scan must cover this new file. No user-controlled URLs, images, or markup are ever interpreted anywhere on this page.
