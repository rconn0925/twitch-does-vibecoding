<fsm_reconciliation>
CHOSEN MINIMAL RECONCILIATION for voting-while-building (surfaced explicitly per amendment — not a silent FSM redesign):

1. **Concurrent rounds, mode untouched.** `RoundManager.startRound()` opens from `IDLE` (owns the mode: transitions IDLE→VOTING_ROUND / VOTING_ROUND→IDLE exactly as today) OR from `BUILD_IN_PROGRESS` (a *concurrent* round: NO mode transitions at open or close — `closeRound()`'s existing `if mode === VOTING_ROUND → IDLE` conditional already handles this). Any other mode still throws `RoundStartError("not-idle")`.
2. **Pill precedence.** The overlay pill stays `PILL_BY_MODE[machine.mode]` — BUILDING wins the pill during a concurrent round while the vote panel renders from the round snapshot. The six-word pill vocabulary is UNCHANGED (no UI-SPEC vocabulary amendment needed; guidance text is panel copy, not a pill word).
3. **One explicit transition-table addition.** `IDLE: [..., "BUILD_IN_PROGRESS"]` in stream-mode.ts TRANSITIONS — used solely by the vote-queue drain (a queued winner starting after the previous build returned to IDLE). Comment it with this rationale.
4. **recordVote decouples from mode.** A vote counts whenever a round is open, not frozen, and mode !== "HALTED" (frozen already covers halt — the mode check is defense-in-depth). Replaces the `mode !== VOTING_ROUND` refusal.
5. **Halt recovery × frozen rounds — full coverage of all three recovery actions (CHOSEN BEHAVIOR).** halt.ts `resume` recovers to `frozen.mode`, which for a halt during a concurrent round is `BUILD_IN_PROGRESS` — a target RoundManager's recovery listener does not handle today, which would leave the round frozen forever and (via the CR-01 round-active guard) wedge every future startRound. FIX in round.ts's prev-HALTED listener: **HALTED→BUILD_IN_PROGRESS with a frozen round → RESUME the frozen round** (`#resume()` re-arms the frozen remainder; acknowledged votes are kept — D2-14 spirit; recordVote already works under BUILD_IN_PROGRESS per point 4). HALTED→VOTING_ROUND → resume (unchanged). HALTED→IDLE (reset-to-idle AND discard-and-resume's BUILD→IDLE mapping) → discard/repool (unchanged `#discard()`). Result: no recovery target can strand a frozen round.
6. **Drain never fires out of a halt (Warning-3 chosen behavior).** The vote-queue drain skips IDLE entries whose PREVIOUS mode was HALTED (D-04 ethos: nothing auto-resumes out of a kill switch — the streamer exits halt into a calm suggestion window, never an instant build). Queued winners resume building at the next round close or manual round start, both of which start the QUEUE HEAD (FIFO preserved; nothing strands as long as the loop runs — and while the loop is deliberately paused, a stranded winner is the streamer's explicit choice, drained by the next manual start).
7. **Manual round start during window/chaos builds is INTENDED (Warning-4).** round.ts cannot (and should not) know which subsystem holds BUILD_IN_PROGRESS; a manual start during a paid-window or chaos build opens a concurrent round whose winner simply QUEUES — the drain skips while the window is live / chaos is on, so the winner builds only after that subsystem finishes and the loop returns to IDLE via a later round close or manual action. Consequence accepted and documented in the console button's title text; no extra route gating.
</fsm_reconciliation>
