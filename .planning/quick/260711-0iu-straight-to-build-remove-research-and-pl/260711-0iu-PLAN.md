<workspace_design>
**How persistence works (amendment A/B) — investigated, minimal-risk design:**

- **Where builds run today:** `sandbox-process.ts` (UNTOUCHED, debug-session-owned) already maps a POSIX-absolute `SpawnOptions.cwd` to `wsl --cd <cwd>`, else falls back to `~` (line ~184; tested at sandbox-process.test.ts:233-245). Nothing anywhere wipes the distro filesystem per task — `sandboxAdapter.terminate()` only runs `wsl.exe --terminate`, which stops the VM; files persist (verified: no rm/reset/wipe path exists in src/). So persistence is the DEFAULT; what's missing is a STABLE, ROTATABLE path and mode awareness.
- **Design — generation rotation, not deletion:** the workspace dir is `/home/builder/projects/app-<generation>` (POSIX absolute inside the distro). "New project" INCREMENTS the generation — the next build cds into a fresh empty dir and scaffolds; the previous dir stays on disk untouched (archive-by-construction; zero destructive commands, zero new wsl.exe exec surface, zero new deps).
- **State:** a new single-row `workspace_state` SQLite table (generation, scaffolded flag) — survives process crash mid-stream (same durable-state rationale as the task queue). `scaffolded` flips to 1 ONLY when a build finalizes `done` in the current generation: a failed/refused/vetoed first build leaves the next attempt in scaffold mode (nothing meaningful was built).
- **Threading:** `AgentRunSpec` gains `workspaceDir: string`; sdk-runner sets `options.cwd = spec.workspaceDir` on the sandboxed branch. The Agent SDK forwards `Options.cwd` into the `SpawnOptions` given to `spawnClaudeCodeProcess`, which sandbox-process already translates. **Executor MUST verify** in `node_modules/@anthropic-ai/claude-agent-sdk` type declarations that `Options.cwd` exists and `SpawnOptions.cwd` receives it (plan-checker confirmed this handoff is real); if a future SDK bump breaks it, the fallback is `~` (builds still persist, generations don't separate) — in that case STOP and surface the finding; do not improvise a sandbox-process edit.
- **Live-verification note:** fakes cannot prove the real WSL2 cwd handoff; add a one-line item to the Phase 5 dry-run expectations in the SUMMARY (first build lands in /home/builder/projects/app-1, second build continues it, New project rotates to app-2).
