<objective>
Chat command layer A — tier-1 voted commands core. `!build <idea>` (new-project intent) and `!revert`/`!undo` (undo-last-change intent) become kind-tagged candidates that enter the SAME pool, the SAME single-gate funnel, and compete in the SAME vote round as `!suggest`. A kind router at build-queue drain executes the winner: suggest → continue (unchanged), build → ship-then-rotate-then-scaffold, revert → mirror git-revert + republish. No `!fork`.

**Locked user decision (mid-planning directive):** whenever chat votes a NEW PROJECT, the current project is committed and pushed FIRST, and rotation happens ONLY on a confirmed publish success — a failed final push keeps the current project active (amber-narrated, audited), never a silent rotate.

Purpose: chat can now agree to start a fresh project or undo the last change by majority vote — not just tweak the current app — without any new classifier path or any destructive filesystem operation.
Output: extended parser/dispatch/narration, `revertLast()` on the gallery publisher, kind router in `drainVoteQueue`, audit + build_history vocabulary, tests for all routing paths.
</objective>
