# Quick Task 260716-h73: Pending Free-Reign Window (Donation Mid-Build) — Summary

**One-liner:** A donation/redemption arriving while the machine is busy (mid-build / mid-round / chaos) now BANKS a durable PENDING window — audited (`window_pending`), narrated immediately, cooldown stamped at grant — that auto-opens with its FULL paid duration the moment the machine returns to IDLE, deterministically beating the auto-cycle's parked/next vote round; HALTED still denies outright and a halt discards a pending window on recovery.

## What Was Built

### Task 1 — FSM pending slot + persistence + audit (`043d775` RED, `890cbf6` GREEN)
- `WindowStatus` widened to `'pending' | 'active' | 'expired' | 'revoked'`; `ControlWindowSnapshot.pending?: true` (console-only flag, `endsAtMs: 0` no-deadline sentinel, documented as never reaching the public wire).
- `insertWindow` takes optional `status` (absent → `'active'`, existing call sites byte-compatible); new `readPendingWindow` + `promotePendingWindow` (rewrites status/opened_at_ms/ends_at_ms — the single durable pending→active transition).
- `open()` guard order: active → pending → HALTED → cooldown → (busy? BANK : open). The bank inserts a pending row, stamps the donor cooldown NOW, writes `window_pending`, emits `WINDOW_PENDING`, returns the pending snapshot — no machine transition, no timer. The D-04 mapping (incl. WR-02 non-USD floor) is computed by the same unchanged code path.
- Promote on IDLE via the STATE_CHANGED handler (`#openFromPending`: null-pending-first re-entrancy guard, full duration from promote-time, `recordWindowOpened` with priorMode IDLE, `WINDOW_OPENED` emitted with `{ fromPending: true }`).
- Halt recovery to IDLE ALSO discards a pending (`#discardPending`: row `'revoked'` + `window_revoked` + `WINDOW_REVOKED` with the pending snapshot); `revoke()` cancels a pending when no active window exists; `restore()` reads pending rows (IDLE → open now with full duration; busy → in-memory pending; promoted rows restore as active — never double).

### Task 2 — Narration, scheduler priority, console pending panel (`f65fbec` RED, `9954594` GREEN)
- Five Narrator beats with the reviewed verbatim copy: `windowPendingDonation`, `windowPendingChannelPoints`, `windowOpenedFromPending` (announces both `!build` and `!suggest`), `windowDeniedPending`, `windowPendingCancelled`. All join the paid copy-separation scan (no chance words).
- main.ts: pending branches in `openWindowFromDonation/Redemption` (no 30s arm at bank), `window-pending` catch → denial beat, `WINDOW_OPENED` fromPending handler (subscribed after `restore()` — amendment 4), `WINDOW_REVOKED` handler branches on `snap.pending` → cancelled beat.
- `isControlWindowLive` widened with `pendingSnapshot() !== null` (scheduler parks through a pending; both race mechanisms commented at the closure).
- Console seam: `snapshot: () => controlWindow.snapshot() ?? controlWindow.pendingSnapshot()` + `on` hook; server subscribes `WINDOW_PENDING`/`WINDOW_REVOKED` → `pushState()` (amendment 2 — banks/discards happen with no mode change; ws-push proven by test). `ConsoleControlWindowSource` gains optional `on?()` mirroring `ConsoleAutoCycleSource`.
- console.js `renderWindowPanel`: pending branch renders "Free-reign window — PENDING" + "Opens the moment the current build/round finishes — full m:ss starts then" (never a countdown from the 0 sentinel); donor line, mapping line, and the SAME Revoke button retained. textContent-only via the existing `el()` helper.
