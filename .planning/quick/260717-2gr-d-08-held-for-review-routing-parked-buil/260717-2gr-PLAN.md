<pinned_decisions>
Planner-pinned resolutions of the "planner decides" items (all other design decisions
in the task directive are LOCKED as written):

P1. **Park lives in main.ts.** build-session gains a park-EXIT path (clean machine
    exit + widened hook) but holds NO parked registry. main.ts owns
    `parkedReviews: Map<reviewId, { task: QueuedTask, phase, provenance, expiresAtMs, timer }>`
    — the branded QueuedTask object is retained there for the continuation dispatch,
    so no re-branding and no gate.ts edits ever happen (single-funnel rail).

P2. **Parked task is REMOVED from the queue at park time** (does NOT keep head
    position). This is how "parked build does not block the queue" is satisfied with
    ZERO t1n startedVoteHeads interaction: drain can never re-pick it, the next vote
    round proceeds normally, and approve dispatches the retained task directly through
    dispatchBuild — p-queue concurrency-1 serializes it behind any in-flight build,
    which is effectively stronger than head position.

P3. **Continuation goes through dispatchBuild** (locked rail): `dispatchBuild(task,
    provenance, opts)` → `buildSession.startBuild(task, provenance, opts)` — still
    exactly ONE `buildSession.startBuild(` call site (grep gate green).
    `opts.resume.phase`:
    - `"mid-build"`: FULL pipeline including the pre-build re-screen (no screening
      exemption — locked decision 3; the suggestion text already passed it once, so a
      re-pass is expected; a fresh hold re-parks and that is correct). The build
      prompt gains a HOST-AUTHORED approved-continuation note.
    - `"pre-build"`: skips screenBuildPlan ONLY — the streamer's approval IS the
      escalation resolution for that exact text (same doctrine as intake approve(),
      which re-pools with rationale "Approved by streamer review" and never
      re-classifies). Without this, re-dispatch re-holds the same text in an infinite
      park loop. In-flight screening stays FULLY active either way.

P4. **Preview taint is fail-closed for ANY mid-build hold** (we cannot know whether
    the flagged Write executed before the abort — assume it did). Mechanism composing
    with 093: the park path does NOT fire fireTeardownHook; main.ts's onHeldForReview
    handler owns preview semantics instead:
    - project-switch hold (previewHoldoverGeneration !== null) → raw reRootPreview()
      (holdover-aware → previous project back on screen). Holdover NOT discharged.
    - tweak hold (holdover null) → set `previewParked = true`, NO reroot — the
      sandbox terminate already killed the dev server, so the 4902 page shows the
      existing standing-by card (quick-ofs). Withholding the 093 resurrection IS the
      park mechanism; nothing new to stop.
    - pre-build hold → no preview action at all (no teardown happened).
    Discharge of previewParked: (a) approve of the tweak hold → clear + reroot
    immediately (content now sanctioned; mid-build preview serves the dir normally);
    (b) next successful done build → onBuildDone clears + reroots (mirrors holdover
    discharge); (c) any operator immediate path → rerootPreviewNow also clears it.
    Reject/expire of a tweak hold KEEPS the park (simpler safe option — no file
    revert). Guard BOTH raw-seam reroot subscribers (onBuildTeardown handler and the
    093 HALTED-exit reroot) with `previewParked && previewHoldoverGeneration === null`
    → skip, so a later teardown/recovery can never resurrect the tainted dir.

P5. **Expiry = 120s auto-decline** (AMENDMENT): `REVIEW_HOLD_TIMEOUT_SECONDS` via
    envPositive, default 120. Per-item unref'd setTimeout armed when the review row is
    inserted (the hold moment). On fire: expireOne() (review_expired audit) → discard
    the parked entry → comp02Rejected denial narration → preview park unchanged →
    console push. The existing 4h REVIEW_TTL_HOURS intake TTL and 15-min sweep are
    UNTOUCHED (they govern intake holds; boot-time expireAllPending already covers
    crash restore of parked rows — fail-closed).

P6. **No schema migration.** review_queue columns as-is: suggestion_text = the TASK
    text (identity); rationale = classifier rationale, plus for mid-build holds a
    bounded flagged-content excerpt appended (first 1500 chars, separator line) —
    console-only surface (WR-01 origin-checked ws + CSRF routes already protect it;
    never on overlay/chat). New audit event_type string `build_parked_for_review`
    (audit_log has no CHECK constraint — the round_opened initiator precedent).

P7. **Narration:** park reuses the existing buildHeld beat; reject AND expiry reuse
    comp02Rejected (the existing denial beat, locked decision 6); approve-resume adds
    ONE new BuildNarrator method `buildResumedFromReview(title)` (types.ts +
    ingestion/narration.ts + test fakes) — calm beat, e.g. "streamer approved — picking
    it back up". Never silent (locked rail 8).

P8. **HALTED semantics untouched:** approve while HALTED → heldBuilds.resolve returns
    "conflict" → 409 (checked BEFORE resolving the row). Parks survive a halt (they
    are not active work; abortActiveWork never touches them); the 120s timer keeps
    running through a halt (fail-closed decline is acceptable while halted — db.open
    guards writes). Halt recovery does not auto-resume anything.

Accepted edge (document in SUMMARY, no code): if the workspace rotates during the
120s park window (another switch build shipping within 2 minutes — build durations
make this near-impossible), the continuation builds in the CURRENT workspace; the
prompt says "continue from the current workspace state", content already approved.
